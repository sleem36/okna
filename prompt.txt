
Ты — опытный full-stack разработчик, специализирующийся на Next.js, парсинге данных и миграции с WordPress. Помоги мне перенести существующий сайт на WordPress в статический прототип на Next.js. Сайт находится локально (http://poli.local/, файлы в C:\ospanel\domains\poli.local).

Исходные данные
XML-экспорт всего содержимого WordPress: файл WordPress.2026-03-11.xml (около 150 000 строк, содержит записи, страницы, рубрики, метки, меню, произвольные поля).

Медиафайлы: папка uploads из wp-content (копия доступна).

Целевой стек: Next.js (последняя версия), CSS-модули / Tailwind (на твой выбор), функциональные компоненты.

Будущее: после этого этапа подключим Strapi в качестве бэкенда, поэтому данные должны быть подготовлены для лёгкой замены на API-запросы.

Задача — разбить на подзадачи, для которых ты будешь генерировать код и инструкции
1. Подготовка данных (парсинг XML → JSON)
Напиши скрипт (Node.js с библиотекой xml2js или подобной), который:

Прочитает WordPress.2026-03-11.xml.

Извлечёт все записи (post_type = 'post') и страницы (post_type = 'page') в отдельные JSON-файлы (posts.json, pages.json).

Для каждого элемента сохрани поля:

id (WP post ID)

title

slug (post_name)

content (очищенный от лишних тегов, но с сохранением базового HTML)

excerpt

date

featured_image (URL из метаполя _thumbnail_id, связать с вложением)

categories (массив слагов/названий)

tags

Извлеки меню из раздела wp:nav_menu (обычно там элементы с wp:menu_item). Создай menus.json с иерархической структурой (пункты, подпункты).

Извлеки рубрики (category) и метки (tag) в отдельные JSON.

Все ссылки на медиафайлы замени на пути вида /uploads/... (соответствует будущему размещению в public).

Скрипт должен быть самодостаточным, с комментариями и обработкой ошибок. Выдай мне его полный код и инструкцию по запуску.

2. Создание Next.js-приложения (статический прототип)
Инициализируй проект Next.js (App Router или Pages Router — выбери оптимальный для SEO и статической генерации, объясни почему).

Организуй структуру папок:

data/ — для JSON-файлов, полученных на шаге 1.

components/ — переиспользуемые компоненты (Header, Footer, Menu, Card, Layout и т.д.).

app/ или pages/ — маршруты, соответствующие URL исходного сайта (например, /, /[slug] для страниц, /category/[slug] для рубрик, /post/[slug] для записей).

Напиши компоненты для отображения данных из JSON:

Главная страница: должна выводить последние записи/карусель, как в оригинале.

Страница записи/поста: выводить заголовок, дату, контент, изображение, рубрики.

Страница рубрики: список записей этой рубрики.

Меню: должно строиться из menus.json и быть интерактивным (возможно, с выпадающими списками).

Вёрстка:

Полностью скопируй дизайн текущего сайта (я могу предоставить скриншоты или описать, но ты должен ориентироваться на структуру, заложенную в XML, и на типичные элементы WordPress-тем). Используй CSS-модули или Tailwind для стилизации.

Адаптивность: мобильная, планшетная, десктопная версии.

Интерактивность: если в исходном сайте есть бургер-меню, слайдеры, табы, аккордеоны — реализуй их на чистом JavaScript/React без jQuery.

Производительность и SEO:

Используй next/image для оптимизации изображений.

Подключи локальные шрифты через next/font.

Добавь мета-теги (title, description) в каждую страницу через next/head (заглушки, можно взять из данных).

3. Интеграция с будущим Strapi
Продумай структуру JSON так, чтобы позже можно было легко заменить импорт данных на fetch-запросы к Strapi (например, поля называть так же, как в API).

В компонентах используйте пропсы для данных, чтобы они были независимы от источника.

4. Документация и запуск
Напиши README.md с шагами: как установить зависимости, запустить скрипт парсинга, запустить дев-сервер.

Укажи, какие команды выполнить: npm install, npm run parse (если добавишь скрипт в package.json), npm run dev.

Дополнительные пожелания
Поскольку XML большой, скрипт должен обрабатывать его потоково или с учётом памяти, чтобы не зависнуть.

Все ссылки внутри контента (на другие страницы/записи) должны быть преобразованы в локальные Next.js-ссылки (например, /post/about-us).

Для изображений в контенте (например, в <img>) нужно также заменить пути на /uploads/... и, возможно, обернуть в next/image.
